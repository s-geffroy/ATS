Conformance vectors

The contract that turns ATS from a personal project into a reliable standard. Every conformant implementation produces bit-identical output on these 104 vectors across 9 datasets. Filterable tables, one-click JSON copy, works offline (Service Worker since v0.6).

What conformance vectors are

A conformance vector is a deterministic test case: a UTC instant plus the ATS encoding(s) that an implementation must produce when given that input. The set of vectors below is the executable form of the spec's conformance contract — running them is how a third-party library proves it implements ATS correctly. The normative definition lives in manifesto §16.5 (conformance) and §9.3 (conversion contract); the vector additivity policy lives in versioning §4.

Schema (illustrated by the core file)

{
  "spec_version": "0.7",          // MAJOR.MINOR of the spec that produced these vectors
  "epoch_utc": "1969-07-20T00:00:00Z",
  "precision": 5,                  // default fractional digits (manifesto §4.4)
  "vectors": [
    {
      "label": "Epoch (start of Apollo 11 landing day)",
      "utc": "1969-07-20T00:00:00Z",
      "canonical": "T+ Δ 0.0.0.0.00000",
      "display": "Δ0.0.0.0-00.0",
      "sign": "T+",
      "kilo": 0, "hecto": 0, "deka": 0, "kin": 0,
      "frac": 0
    },
    …
  ]
}

The spec_version field is mandatory on every vector file (versioning §2). The arithmetic and bridge files use small variations on this schema — the JS controller below adapts automatically.

How to run them

Pick the reference implementation closest to your stack and adapt:

# Python — covers all 9 datasets bit-for-bit
docker run --rm -v "$PWD:/app" -w /app python:3.13-slim \
  sh -c 'pip install --quiet ".[bridges,tests]" && python -m unittest discover tests'
// JavaScript — covers the Earth core + arithmetic
docker run --rm -v "$PWD:/app" -w /app node:20-slim \
  node tests/test_vectors.mjs
# 12/12 JS vectors pass

A claim of conformance to ATS v0.7 means passing the core (12) and arithmetic (12) sets bit-for-bit (manifesto §16.5). The multi-planetary, bridge sets are optional and must be declared explicitly if claimed.

Why exactly these numbers

  • 12 core — covers the epoch, the touchdown remarkable instant, the four anchors of the day (Bloc 0/5 = midnight/noon UTC), large Kilo (verifies unboundedness), fractional edge cases (frac=99999 limit), and the T- branch.
  • 12 arithmetic — exercises all seven typed Δ/Δd operations from §11.4, the Kin → Deka → Hecto → Kilo carries, the epoch-crossing direction flip, and cross-sign comparisons.
  • 10 per body / bridge — minimum coverage required by multi-planetary §6.3: epoch, round milestones, complex fractions, T-, a 2050+ date, and contemporary instants.

For implementers

  • Vectors are additive only. A published vector is never modified after release (versioning §4.1). If an error is found, the corrected vector is added and the wrong one marked deprecated: true.
  • Spec-version drift. A consumer claiming conformance to vX.Y must reject files declaring spec_version > vX.Y and must pass every non-deprecated vector with spec_version ≤ vX.Y.
  • Forward compatibility (skipping unknown vectors with a warning) is RECOMMENDED but not REQUIRED.
  • Proposing new vectors follows the lightweight RFC procedure in versioning §6 — minimum 14 days of public comment.
